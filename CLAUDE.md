## iOS app (Capacitor)
`ios/` is a Capacitor 7 wrapper around the same `index.html`, **bundled mode** (not hybrid/remote — a `server.url` experiment on 2026-07-19 broke the native plugin bridge in ways described below and was reverted same day). `npm run build:www` copies web files + `capacitor.js` into `www/`; `npm run ios` = build + sync + patch + open Xcode; `npm run ios:sync` does the same without opening Xcode (use this to verify a build compiles). Requires Xcode 16+ (13.x fails compiling Capacitor 7 — Jeff updated 2026-07-20).

**Consequence of bundled mode**: the native app needs an Xcode rebuild+reinstall to pick up any `index.html`/plugin change — not just the weekly re-sign. Since re-signing already requires cable+Xcode+▶ weekly regardless (free Apple ID, 7-day cert expiry — the $99/yr program removes this), that's the natural moment to also grab pending changes; rebuild sooner only if something's urgent. The website/PWA on GitHub Pages stays fully instant — this only affects the native app's own copy. Rebuilding (even after Clean Build Folder) never touches on-device data — only fully deleting the app from the phone does (confirmed the hard way 2026-07-19).

`HealthSyncPlugin.swift` reads weight/kcal/protein/steps from HealthKit and returns rows shaped like the clipboard-import JSON, so the JS merge path is shared. Info.plist has the Health usage strings (read-only, app never writes to Health); `App.entitlements` + pbxproj carry the HealthKit entitlement; the "HealthKit" capability is also added via Xcode's own + Capability UI (Jeff confirmed present 2026-07-20) — belt and suspenders, but the capability really is required for `requestAuthorization` to not silently no-op.

**Three non-obvious plugin-bridge bugs found 2026-07-19/20, all required to get HealthKit working at all — read before touching iOS plugin code again:**
1. `@capacitor/core`'s JS runtime (`registerPlugin`, etc.) is NOT part of the native-injected bridge — it's a separate library every real Capacitor app bundles into its own page. Our single-file `index.html` never did, in *any* loading mode (this, not remote-loading, was the real cause of the "registerPlugin is not a function" saga — the hybrid-mode theory was a red herring). Fixed by copying `node_modules/@capacitor/core/dist/capacitor.js` into `www/` and loading it via `<script src="capacitor.js">` near the top of `index.html` — 404s harmlessly on GitHub Pages/artifact where the file doesn't exist.
2. Capacitor's native iOS runtime (`CapacitorBridge.swift`) registers custom *local* Swift plugins ONLY via a `packageClassList` array read from the bundled `capacitor.config.json` at `Bundle.main` — there is no Objective-C runtime auto-scanning despite `CAPBridgedPlugin` conformance suggesting otherwise. `npx cap sync` recomputes this field from installed npm plugin packages and silently wipes any hand-added entry (our HealthSyncPlugin isn't an npm package, so it's invisible to that scan). `scripts/patch-ios-config.js` re-injects the local plugin name(s) into `ios/App/App/capacitor.config.json` after every sync; wired into `npm run ios` / `npm run ios:sync`. **It appends, never overwrites** — official plugins (Filesystem, Share, ...) DO get correctly auto-added by `cap sync` itself, and an earlier version of this script that overwrote the array would have deregistered them. If a future local Swift plugin is added, add its class name to `OUR_LOCAL_PLUGINS` in that script.
3. **Export/backup on native**: the anchor+download trick used for web export is a silent no-op inside a native WKWebView (no download manager to catch it) — broken the whole time the native app existed; Jeff only discovered it via the data-loss scare below. Fixed with `@capacitor/filesystem` + `@capacitor/share` (Capacitor 7-compatible versions — `@capacitor/filesystem@^8`/`@capacitor/share` need to stay pinned to `^7` until `@capacitor/core` is bumped past 7, or `npm install` will pull an incompatible peer). On native, Export writes the backup JSON via `Filesystem.writeFile` then hands it to iOS's native Share sheet so Jeff can Save to Files/iCloud/AirDrop/Notes; web keeps the working Blob+anchor download.

**Data-loss incident (2026-07-19)**: mid-HealthKit-troubleshooting, Jeff was asked to fully delete+reinstall the app to rule out stale caching — this wiped the native app's on-device storage (expected/correct behavior for a delete, but the *manual* export "safety net" suggested first turned out to be silently broken per bug #3 above). Nothing important was lost only because the PWA on GitHub Pages was the real source of truth and untouched. This is why the backup reminder (below) exists.

- **Auto-sync on native**: Health data pulls automatically on launch and on foreground (`visibilitychange`), throttled to once per 15 min (`cutTracker.lastHealthAutoSync`). Deliberately NOT using HealthKit Background Delivery — needs an observer query + Background Modes capability with no code behind it; foreground auto-sync gets "open it and it's filled in" without that complexity. The Import from Health button still exists for a manual/immediate pull, and is also the button that must be tapped once to trigger Apple's permission sheet the first time.
- **Backup reminder**: `cutTracker.lastBackupAt` (separate localStorage key, deliberately outside `state` — same reasoning as the API key) timestamps the last successful export (native and web both). A status line under the Targets & data buttons shows days-since-backup, amber at 7+ days or if never backed up.

## What the app does
- Daily log (date / weight kg / kcal / protein g) → localStorage under key `cutTracker.v1`. Partial entries fine; fields merge per date.
- Trend chart: raw weigh-ins as faint dots, 7-day trailing average as the line, dashed goal lines at 12% (74 kg) and 10% (72.2 kg). Mark legend below; 1w/1m/All range chips (`config.chartRange`) window the *display only* — regression and averages always use full data.
- Daily log rows: tap to load a day back into the form for editing (same-date submit merges non-null fields); swipe left to reveal Delete (two-step on purpose — the old always-visible × was too easy to fat-finger).
- Seven guardrails over the last 7 logged days, worst one becomes the headline verdict:
  1. **Loss rate** — target band 0.5–0.7 kg/wk; >1.0 is crit.
  2. **Eating your target** — flags *under*-eating vs the 2,300 kcal target. This is the app's reason to exist: Jeff once averaged ~1,700 vs a 2,300 target and lost serious strength (weighted dips 40 kg → bodyweight). Under-target must always be at least as loud as over-target.
  3. **Energy floor** — any single day below RMR (1,964) is crit.
  4. **Muscle risk** — intake-based deficit (maintenance − avg kcal) vs fat-supply ceiling (fatMass × 30 kcal/kg/day ≈ 381).
  5. **Protein** — ≥160 g/day.
  6. **Strength** — recent-21-day best e1RM (Epley) per benchmark lift vs cut-best, worst lift judged: ≥97% good, 92–97% warn, <92% crit. Needs ≥2 sets per lift. This is the outcome the app protects, measured directly — Jeff's 2025 strength collapse (dips) is the reference disaster. Sets live in `state.strength` [{date,lift,w,reps}], logged via the Strength check card (stamped today); benchmark lift names are config lift1–3 (Weighted dip / Weighted pull-up / Squat — Jeff's picks 2026-07-19).
  7. **Activity** — 7-day avg steps vs the 8k–12k band the maintenance estimate assumes. Warn-only (never crit) and deliberately LAST in order: it's a calibration check on the calorie math, not a compliance target, and steps must never enter the deficit calculation (device burn numbers are 20–30% noise — the scale trend is the real expenditure meter).
- Progress bars to both goals with ETA from current rate; settings panel (all targets editable); JSON export/import.
- **Import from Health** button has two implementations sharing one merge path (`mergeHealthRows`/`parseHealthImport`), auto-detected at runtime: on the **native iOS app**, it calls `HealthSyncPlugin` directly (see iOS section above — this is now the primary path and also auto-runs on launch/foreground). On the **web PWA**, no native plugin exists, so it falls back to reading clipboard JSON placed there by Jeff's iOS Shortcut (`{"date":"yyyy-mm-dd","weight":"76.2 kg","kcal":"2,250 kcal","protein":"162 g","steps":"9,800"}`, single object or array). Units/commas stripped; zero kcal/protein skipped (zero = "no Health samples", importing it would fabricate a crit under-eating day). Keep the parse/merge split shared between both paths.

- **Coach's report (AI analysis)**: button builds a compact JSON of targets/trend/guardrails/strength/last-14-days and asks `claude-opus-4-8` (raw fetch to api.anthropic.com with `anthropic-dangerous-direct-browser-access`; no sampling params — they 400 on Opus 4.8). Jeff's API key lives in localStorage `cutTracker.apiKey`, deliberately OUTSIDE state so exports/backups never contain it. No key → the same prompt is copied to the clipboard for pasting into the Claude app. Won't work on the artifact origin (CSP) — GitHub Pages only.

## Non-obvious design decisions (don't silently reverse)
- **Loss rate = linear regression** over the last 21 days of weigh-ins, widening to all data when the window has <2 points or <7 days span. Never day-to-day deltas — daily swings are glycogen/water.
- **Muscle risk uses the intake deficit, not the scale-derived one.** Early-cut water/glycogen losses make kg-on-the-scale × 7,700 kcal wildly overstate the true deficit and would contradict the other cards.
- **Muscle-risk bands are deliberately loose** (good ≤ ceiling+200, warn to +450, crit beyond): Jeff's *planned* 500 kcal deficit slightly exceeds the ~381 fat ceiling. If on-plan reads yellow forever, the signal dies. Keep "on-plan = green".
- Verdict priority is crit > warn > good, first match in guardrail order — deliberate, so under-eating outranks everything else at equal severity.
- Weekly averages over daily numbers, everywhere, in every message shown to Jeff.
- **Visual identity** (2026-07 pass): "clinical instrument" — cool grey-blue neutrals, tabular mono numerals, severity colors. Wordmark + day counter use Avenir Next Condensed via `--display` (ships on Apple platforms; no font files — CSP). The signature element is the seven-segment strip in the verdict hero (RATE·KCAL·FLOOR·MUSCLE·PROTEIN·LIFTS·STEPS — plain words; Jeff couldn't parse cuter abbreviations), one tick per guardrail in card order, tap scrolls to the card — it exists to make the "verdict = worst of seven" logic visible; don't decorate it or reorder it away from the guardrail order.

## Gotchas
- localStorage is per-origin: file://, localhost, GitHub Pages, the artifact URL, and the native iOS app's own container all hold separate data. Jeff's real historical log lives on the GitHub Pages origin (migrated from the artifact 2026-07-19, confirmed intact). The native app is a separate, less-trusted copy — see the iOS section's data-loss incident before assuming its storage is authoritative.
- Storage key is `cutTracker.v1`. If you change the data shape, migrate on load — don't bump the key and orphan his log.
- The page is published as an artifact: strict CSP, no external requests, so it must stay fully self-contained (it already is — keep it that way).
- Viewport is pinned (`user-scalable=no`) and `touch-action: manipulation` is set globally: WebKit's double-tap-to-zoom was breaking the layout on double-taps anywhere in the app (2026-07-19). Don't remove this — a controlled dashboard shouldn't pinch/double-tap zoom, same as a native app wouldn't.
- File starts with `<title>` directly (no doctype/html/head/body) because the Artifact tool wraps it; browsers handle it fine locally too.
