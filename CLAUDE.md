## iOS app (Capacitor)
`ios/` is a Capacitor 7 wrapper around the same `index.html` (`npm run build:www` copies web files to `www/`; `npm run ios` = build + sync + open Xcode). `HealthSyncPlugin.swift` reads weight/kcal/protein/steps from HealthKit and returns rows shaped like the clipboard-import JSON, so the JS merge path is shared; on native, the "Import from Health" button calls the plugin instead of the clipboard. Info.plist has the Health usage strings; `App.entitlements` + pbxproj carry the HealthKit entitlement. Hybrid mode: `capacitor.config.json` sets `server.url` to the GitHub Pages URL, so the shell loads the LIVE site — `git push` updates the app instantly; Xcode is only needed for re-signing or Swift changes. (Trade-off Jeff accepted: weaker offline. Delete the `server` block to go back to bundled `www/`.) **Bundled mode (reverted from hybrid 2026-07-19).** Tried `server.url` pointing at the live GitHub Pages URL for instant native-app updates. Reverted to bundled `www/` (built via `npm run build:www` from the current `index.html`) for reliability. Consequence: the native app now needs an Xcode rebuild+reinstall to pick up any `index.html` change, not just the weekly re-sign — but since the re-sign already requires cable+Xcode+▶ weekly regardless, that's the natural moment to also grab pending changes. The website/PWA on GitHub Pages remains fully instant — this only affects the native app's own bundled copy. Requires Xcode 16+ (13.x fails compiling Capacitor 7).

**Two non-obvious plugin-bridge gotchas found 2026-07-19/20, both required to get HealthKit working at all:**
1. `@capacitor/core`'s JS runtime (`registerPlugin`, etc.) is NOT part of the native-injected bridge — it's a separate library every real Capacitor app bundles into its own page. Our single-file `index.html` never did. Fixed by copying `node_modules/@capacitor/core/dist/capacitor.js` into `www/` (part of `npm run build:www`) and loading it via `<script src="capacitor.js">` near the top of `index.html` — 404s harmlessly on GitHub Pages/artifact where the file doesn't exist.
2. Capacitor's native iOS runtime (`CapacitorBridge.swift`) registers custom local Swift plugins ONLY via a `packageClassList` array read from the bundled `capacitor.config.json` at `Bundle.main` — there is no Objective-C runtime auto-scanning despite `CAPBridgedPlugin` conformance suggesting otherwise. `npx cap sync` recomputes this field from installed npm plugin packages and silently wipes any hand-added entry (our HealthSyncPlugin isn't an npm package, so it's invisible to that scan) — the root `capacitor.config.json`'s `packageClassList` value is NOT what ends up on device. `scripts/patch-ios-config.js` re-injects `["HealthSyncPlugin"]` into `ios/App/App/capacitor.config.json` after every sync; wired into `npm run ios` / `npm run ios:sync`. If a future Swift plugin is added, add its class name to that script too.
- Use `npm run ios:sync` to rebuild the www+iOS project without opening Xcode (for verifying builds); `npm run ios` also opens Xcode.
- **Export/backup on native (2026-07-20)**: the anchor+download trick used for web export is a silent no-op inside a native WKWebView (no download manager to catch it) — this was broken the whole time the native app existed and Jeff only discovered it via a data-loss scare. Fixed with `@capacitor/filesystem` + `@capacitor/share` (Capacitor 7-compatible versions — `@capacitor/filesystem@^8` requires core >=8, don't upgrade past `^7` without bumping core too): on native, Export writes the backup JSON via `Filesystem.writeFile` then hands it to iOS's native Share sheet (`Share.share`) so Jeff can Save to Files/iCloud/AirDrop/Notes; web keeps the working Blob+anchor download. Official plugins (Filesystem, Share) DO get auto-added to `packageClassList` by `cap sync` — `scripts/patch-ios-config.js` only appends `HealthSyncPlugin`, it must never overwrite the array (that bug briefly existed and would have deregistered these).
- **Backup reminder**: `cutTracker.lastBackupAt` (separate localStorage key, deliberately outside `state` — same reasoning as the API key) timestamps the last successful export (both native and web paths). A status line under the Targets & data buttons shows days-since-backup, turning amber/warning at 7+ days or if never backed up. This exists because native-app data has genuinely been lost once already (2026-07-19, mid-troubleshooting) with no way to recover it — the PWA's copy was the only reason nothing important was lost. ## What the app does
- Daily log (date / weight kg / kcal / protein g) → localStorage under key `cutTracker.v1`. Partial entries fine; fields merge per date.
- Trend chart: raw weigh-ins as faint dots, 7-day trailing average as the line, dashed goal lines at 12% (74 kg) and 10% (72.2 kg). Mark legend below; 1w/1m/All range chips (`config.chartRange`) window the *display only* — regression and averages always use full data.
- Daily log rows: tap to load a day back into the form for editing (same-date submit merges non-null fields); swipe left to reveal Delete (two-step on purpose — the old always-visible × was too easy to fat-finger).
- Seven guardrails over the last 7 logged days, worst one becomes the headline verdict:
  1. **Loss rate** — target band 0.5–0.7 kg/wk; >1.0 is crit.
  2. **Eating your target** — flags *under*-eating vs the 2,300 kcal target. This is the app's reason to exist: Jeff once averaged ~1,700 vs a 2,300 target and lost serious strength (weighted dips 40 kg → bodyweight). Under-target must always be at least as loud as over-target.
  3. **Energy floor** — any single day below RMR (1,964) is crit.
  4. **Muscle risk** — intake-based deficit (maintenance − avg kcal) vs fat-supply ceiling (fatMass × 30 kcal/kg/day ≈ 381).
  5. **Protein** — ≥160 g/day.
  6. **Strength** — recent-21-day best e1RM (Epley) per benchmark lift vs cut-best, worst lift judged: ≥97% good, 92–97% warn, <92% crit. Needs ≥2 sets per lift. This is the outcome the app protects, measured directly — Jeff's 2025 strength collapse (dips) is the reference disaster. Sets live in `state.strength` [{date,lift,w,reps}], logged via the Strength check card (stamped today); benchmark lift names are config lift1–3 (Weighted dip / Weighted pull-up / Squat — Jeff's picks 2026-07-19).
  7. **Activity** — 7-day avg steps vs the 8k–12k band the maintenance estimate assumes. Warn-only (never crit) and deliberately LAST in order: it's a calibration check on the calorie math, not a compliance target, and steps must never enter the deficit calculation (device burn numbers are 20–30% noise — the scale trend is the real expenditure meter).
- Progress bars to both goals with ETA from current rate; settings panel (all targets editable); JSON export/import.
- **Auto-sync on native**: when running as the Capacitor iOS app, Health data pulls automatically on launch and on foreground (visibilitychange), throttled to once per 15 min (`cutTracker.lastHealthAutoSync`). Deliberately NOT using HealthKit Background Delivery — that needs an observer query + Background Modes capability with no code behind it yet; foreground auto-sync gets "open it and it's filled in" without that complexity. The button still exists for a manual/immediate pull.
- **Import from Health**: reads clipboard JSON put there by Jeff's iOS Shortcut (`{"date":"yyyy-mm-dd","weight":"76.2 kg","kcal":"2,250 kcal","protein":"162 g","steps":"9,800"}`, single object or array). Units/commas stripped; zero kcal/protein skipped (zero = "no Health samples", importing it would fabricate a crit under-eating day). Merges per date like manual entry. This is the stepping stone to a Capacitor+HealthKit native app (Jeff's stated goal) — keep the parse/merge split so native can reuse it.

- **Coach's report (AI analysis)**: button builds a compact JSON of targets/trend/guardrails/strength/last-14-days and asks `claude-opus-4-8` (raw fetch to api.anthropic.com with `anthropic-dangerous-direct-browser-access`; no sampling params — they 400 on Opus 4.8). Jeff's API key lives in localStorage `cutTracker.apiKey`, deliberately OUTSIDE state so exports/backups never contain it. No key → the same prompt is copied to the clipboard for pasting into the Claude app. Won't work on the artifact origin (CSP) — GitHub Pages only.

## Non-obvious design decisions (don't silently reverse)
- **Loss rate = linear regression** over the last 21 days of weigh-ins, widening to all data when the window has <2 points or <7 days span. Never day-to-day deltas — daily swings are glycogen/water.
- **Muscle risk uses the intake deficit, not the scale-derived one.** Early-cut water/glycogen losses make kg-on-the-scale × 7,700 kcal wildly overstate the true deficit and would contradict the other cards.
- **Muscle-risk bands are deliberately loose** (good ≤ ceiling+200, warn to +450, crit beyond): Jeff's *planned* 500 kcal deficit slightly exceeds the ~381 fat ceiling. If on-plan reads yellow forever, the signal dies. Keep "on-plan = green".
- Verdict priority is crit > warn > good, first match in guardrail order — deliberate, so under-eating outranks everything else at equal severity.
- Weekly averages over daily numbers, everywhere, in every message shown to Jeff.
- **Visual identity** (2026-07 pass): "clinical instrument" — cool grey-blue neutrals, tabular mono numerals, severity colors. Wordmark + day counter use Avenir Next Condensed via `--display` (ships on Apple platforms; no font files — CSP). The signature element is the seven-segment strip in the verdict hero (RATE·KCAL·FLOOR·MUSCLE·PROTEIN·LIFTS·STEPS — plain words; Jeff couldn't parse cuter abbreviations), one tick per guardrail in card order, tap scrolls to the card — it exists to make the "verdict = worst of seven" logic visible; don't decorate it or reorder it away from the guardrail order.

## Gotchas
- localStorage is per-origin: file://, localhost, GitHub Pages, and the artifact URL hold separate data. Jeff's real data belongs on the GitHub Pages origin (migration from the artifact via Export/Import was pending as of 2026-07-19 — confirm it happened before assuming).
- Storage key is `cutTracker.v1`. If you change the data shape, migrate on load — don't bump the key and orphan his log.
- The page is published as an artifact: strict CSP, no external requests, so it must stay fully self-contained (it already is — keep it that way).
- Viewport is pinned (`user-scalable=no`) and `touch-action: manipulation` is set globally: WebKit's double-tap-to-zoom was breaking the layout on double-taps anywhere in the app (2026-07-19). Don't remove this — a controlled dashboard shouldn't pinch/double-tap zoom, same as a native app wouldn't.
- File starts with `<title>` directly (no doctype/html/head/body) because the Artifact tool wraps it; browsers handle it fine locally too.
